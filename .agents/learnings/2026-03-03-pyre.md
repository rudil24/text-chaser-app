# Learning: Qt eventFilter must guard by event type

__ID__: L1
__Category__: debugging
__Confidence__: high

## What We Learned

`QObject.installEventFilter` receives ALL Qt event types by default — KeyPress, KeyRelease, MouseMove, Paint, etc. Failing to add `event.type() == QEvent.Type.KeyPress` causes key release events to be processed as keystrokes, silently multiplying counter increments (in PYRE: doubled the backspace count, burning at ~5 presses instead of 10).

## Why It Matters

Any PyQt6/PySide6 feature involving keyboard interception via event filters will have subtle, hard-to-reproduce bugs if event type is not checked explicitly. The bug manifests as inconsistent behavior that varies by typing speed.

## Source

PYRE `editor_screen.py` backspace counter bug — found in manual testing, fixed by adding `QEvent.Type.KeyPress` guard.

# Learning: State machine finalizers need a bypass write path

__ID__: L2
__Category__: debugging
__Confidence__: high

## What We Learned

When a `_finalized` flag guards a `save()` method to prevent timer-driven writes after session end, the finalizer methods (`finalize_burn`, `finalize_clean`) must NOT call `save()` — they set `_finalized = True` before writing, causing `save()` to bail immediately. The fix: extract a private `_write()` method with no guard, called directly by finalizers. The public `save()` method calls `_write()` only after passing the guard.

## Why It Matters

This pattern (state flag + periodic write + explicit finalization) is common in any app with auto-save semantics. The self-block bug causes silent data loss — the app appears to work but files are never written.

## Source

PYRE `auto_save.py` — `finalize_burn`/`finalize_clean` wrote no files to disk. Found by Vera's unit tests.

# Learning: HTML mockups are viable pixel-ready design tools for desktop apps

__ID__: L3
__Category__: process
__Confidence__: high

## What We Learned

Self-contained HTML files with real CSS animations, Google Fonts, and accurate color palettes can serve as "pixel-ready" design presentations for desktop apps — viewable directly in the browser with no external tools. Product Owner validated: "mockups in html worked really well, I was skeptical." The format supports multiple states (start/active/deletion) on one scrollable page, labeled as design spec panels.

## Why It Matters

Eliminates the Figma dependency for UI approval gates on projects where the PO is comfortable in a browser. Stella can produce these in a single subagent call. The approval gate (see design → choose → lock → build) is preserved without requiring external tooling.

## Source

PYRE design phase — 3 mockups (DEADTYPE, PYRE, STILL) generated as parallel subagent tasks, presented to PO, decision made in one round.

# Learning: Write tests alongside code, not after

__ID__: L5
__Category__: testing
__Confidence__: high

## What We Learned

The `auto_save.py` self-block bug (L2) shipped to first commit because tests were written after the build phase, not during. Had `test_finalize_burn_writes_file` been written while `finalize_burn` was being implemented, the bug would have been caught in the same session before any commit.

## Why It Matters

Post-hoc testing catches bugs that already shipped. Inline testing catches bugs before they leave the developer's hands. For OPST: Vera should be spawned as a parallel agent during Phase 3/4 to write tests as Stella writes implementation.

## Source

PYRE QA phase — 3 test failures, all in `auto_save.py`, all from a bug in code written in the prior phase.

# Learning: Mock at construction time for path-resolving objects

__ID__: L6
__Category__: testing
__Confidence__: high

## What We Learned

When a class resolves a configurable path in `__init__` (e.g., `AutoSaveManager` calls `get_save_folder()` at construction to build `self._draft_path`), the mock must be active at the moment the object is instantiated — not just during the test body. Patching after construction is too late; the real path is already baked in.

## Why It Matters

A common testing mistake: patching a config function at test scope when the path was already resolved at object creation. Results in tests that appear isolated but write to the real filesystem.

## Source

PYRE `tests/test_auto_save.py` — `_make_manager()` helper uses `with patch(...):` around the `AutoSaveManager(...)` constructor call.

# Learning: Source code must include block-level comments for readability

__ID__: L7
__Category__: process
__Confidence__: high

## What We Learned

Product Owner expectation (confirmed in PYRE retro): all source files must include block-level comments explaining what each class, method, and non-obvious logic block does — documented at a level a junior developer could follow. This was missing from PYRE's source. It is a standing requirement, not optional polish.

## Why It Matters

Code without comments creates onboarding friction and reduces the PO's ability to audit or hand off the project. OPST projects are built to be maintainable, not just functional.

## Source

PYRE retro PO feedback: "would like to see more block-level commenting in the source... i always want documented code that a newby can follow."

# Learning: QA subagent requires Bash permissions to close the testing loop

__ID__: L8
__Category__: process
__Confidence__: high

## What We Learned

Vera (QA) could write tests but could not run them because Bash was not permitted in her subagent context. She had to hand execution back to Cap, breaking the autonomous testing loop. For QA agents to be self-sufficient, they need Bash access scoped to `pytest` and `pip install` within the project venv.

## Why It Matters

An agent that can write tests but not run them is only half-useful. The feedback loop (write → run → fix → re-run) requires execution capability.

## Source

PYRE retro — Vera returned test files but not results; Cap ran the suite manually.
